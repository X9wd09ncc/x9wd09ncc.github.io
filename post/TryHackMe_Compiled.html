漏洞分析报告：Compiled-1688545393558.Compiled

1. 基本信息

属性 值
文件名 Compiled-1688545393558.Compiled
文件格式 ELF 64-bit LSB PIE executable, x86-64
编译器 GCC 11.3.0 (Debian)
链接方式 动态链接 (libc.so.6)
符号信息 未剥离 (not stripped)
入口点 0x1080
源文件 zzz.c

2. 安全缓解措施评估

缓解措施 状态 说明
PIE 启用 ELF 类型为 DYN，地址随机化
NX (栈不可执行) 启用 GNU_STACK 标志为 RW (0x6)，无 X 位
RELRO 部分 GNU_RELRO 存在但仅覆盖部分区域
Stack Canary 未启用 无 __stack_chk_fail 符号，main 函数中无 canary 检查
FORTIFY_SOURCE 未启用 使用 scanf 而非边界检查版本，无编译期加固

风险评级：虽然启用了 PIE 和 NX，但缺少栈金丝雀（Stack Canary）是最严重的防御缺失，使得缓冲区溢出利用几乎不受阻碍。

3. 漏洞清单

3.1 漏洞一：硬编码密码 — 逆向提取（高危）

类型：CWE-798 — 硬编码凭证

位置：.rodata 段地址 0x201f，反汇编 main 函数 0x11d5

详细分析：

密码 _init 以明文硬编码在二进制中。通过 strings 命令即可提取所有候选字符串，再通过反汇编追踪 strcmp 比较目标即可确认。

验证逻辑如下：

main 函数中三次 strcmp 调用的比较对象：
  地址 0x11d5: strcmp(buf, "__dso_handle")  — 第一次过滤
  地址 0x11ef: strcmp(buf, "__dso_handle")  — 第二次过滤
  地址 0x1209: strcmp(buf, "_init")          — 最终验证

简化的 C 逻辑：

char buf[32];
scanf("DoYouEven%sCTF", buf);

// 过滤: buf 不能等于 "__dso_handle" (几乎总是成立)
if (strcmp(buf, "__dso_handle") != 0) {
    if (strcmp(buf, "_init") == 0) {
        puts("Correct!");      // 密码正确
    } else {
        puts("Try again!");
    }
} else {
    puts("Try again!");
}

密码：_init，完整输入为 DoYouEven_init CTF（或等待程序直接比较）

内存中还存在一个干扰字符串 "StringsIsForNoobs"（地址 0x1171-0x118d），它被写入栈上 rbp-0x40 但从未参与任何比较，纯粹是误导。

影响：任何获取到该二进制文件的人都可以在数秒内提取密码，认证机制形同虚设。

3.2 漏洞二：scanf 格式字符串设计缺陷（中危）

类型：CWE-20 — 输入验证不当

位置：main 函数 0x11bd-0x11cc

详细分析：

scanf 使用的格式字符串为 "DoYouEven%sCTF"（地址 0x200f）。

%s 读取非空白字符，直到遇到空白字符或输入耗尽。由于 "CTF" 不是空白字符，%s 会无差别地吞掉它。这导致格式字符串永远无法正确匹配：

输入 %s 提取内容 格式剩余 结果
DoYouEven_initCTF _initCTF 输入耗尽 匹配失败，buf=_initCTF（错误）
DoYouEven_init CTF _init CTF 匹配失败（’ ‘≠’C’）

这意味着程序的输入解析本身具有逻辑缺陷。正确的写法应为 scanf("DoYouEven%31sCTF", buf) 并配合不同的验证方式，或使用分步读取。

影响：密码验证不仅可被绕过（通过逆向提取密码），而且输入格式设计错误可能导致非预期的程序行为。

3.3 漏洞三：栈缓冲区溢出（严重）

类型：CWE-120 — 经典缓冲区溢出

位置：main 函数 0x11cc

详细分析：

栈帧布局（sub $0x40, %rsp，共 64 字节）：

高地址  rbp+0x08  ← 返回地址 (8 bytes)
        rbp+0x00  ← saved RBP  (8 bytes)
        rbp-0x1f  ← 用户输入缓冲区 (32 bytes, scanf %s)
低地址  rbp-0x40  ← 干扰字符串区 (32 bytes, "StringsIsForNoobs")

攻击负载结构：

DoYouEven + [32字节填充] + [8字节伪造RBP] + [8字节目标地址] + CTF
           └── 缓冲区 ──┘   └── 覆盖saved RBP ──┘ └─ 覆盖返回地址 ──┘
  • 溢出距离：32 字节（缓冲区）+ 8 字节（saved RBP）= 40 字节 到达返回地址
  • 不受限写入：scanf("%s") 无宽度限制，可写入任意长度
  • 无 Canary 保护：不会触发 __stack_chk_fail，溢出畅通无阻

利用路径：

尽管启用了 NX 和 PIE，仍存在以下利用方式：

  1. ret2libc：泄漏 libc 基址后，将返回地址覆盖为 system("/bin/sh")，因为程序已加载 libc 且 GOT 表可读
  2. Partial overwrite：仅覆盖返回地址的低 1-2 字节，在 PIE 下跳转到 main 内部其它 gadget，无需完整地址
  3. ret2plt：复用 PLT 中的已有函数（如 printf@plt），构造信息泄漏后二次利用
  4. 栈迁移：若输入写入干扰字符串区域（低 32 字节），可通过伪造 RBP 将栈迁移到可控区域

影响：攻击者可获得任意代码执行能力，完全控制程序流程。这是所有漏洞中最严重的。

4. 严重性总结

漏洞 CWE 严重程度 CVSS 预估
硬编码密码 CWE-798 高 7.5
scanf 格式缺陷 CWE-20 中 5.3
栈缓冲区溢出 CWE-120 严重 9.8

5. 修复建议

  1. 栈缓冲区溢出：将 scanf("%s", buf) 改为 scanf("%31s", buf)，添加宽度限制。同时启用 -fstack-protector-strong 编译选项启用栈金丝雀。

  2. 硬编码密码：移除硬编码凭证，改用基于哈希的密码验证或外部认证机制。

  3. 输入格式缺陷：重新设计输入解析逻辑，使用 fgets + sscanf 或正则匹配，避免 scanf 格式字符串的歧义。

  4. 编译加固：建议添加以下编译参数：

    -fstack-protector-strong -D_FORTIFY_SOURCE=2 -Wl,-z,relro,-z,now
