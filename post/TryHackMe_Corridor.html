IDOR 漏洞渗透测试报告

一、漏洞概述

项目 内容
漏洞名称 不安全的直接对象引用 (IDOR)
漏洞编号 CWE-639
目标系统 http://10.49.161.165
发现日期 2026-05-25
风险等级 高危 (High)
CVSS 评分 7.5 (CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N)
影响范围 可未授权访问隐藏的敏感资源

二、漏洞描述

目标系统是一个名为 “Corridor” 的 Web 应用。主页呈现一条走廊图片，包含 13 扇门（房间 1-13），用户通过点击图片热点区域进入对应房间。每个房间的 URL 端点使用数字的 MD5 哈希值标识。

系统对资源访问控制存在缺陷：服务器未校验用户是否被授权访问特定房间，攻击者可通过枚举 MD5 哈希值访问页面中未引用的隐藏房间（房间 0），从而获取敏感信息（Flag）。

三、漏洞发现过程

3.3 IDOR 利用

识别出哈希规律后，计算数字 0 的 MD5 哈希：

MD5("0") = cfcd208495d565ef66e7dff9f98764da

直接构造并访问未在页面中引用的 URL：

GET http://10.49.161.165/cfcd208495d565ef66e7dff9f98764da

服务器返回了隐藏房间的内容，页面中直接渲染了敏感信息：

<h1>
    flag{2477ef02448ad9156661ac40a6b8862e}
</h1>

隐藏房间 (0) /cfcd208495d565ef66e7dff9f98764da Flag 内容

四、漏洞根因分析

  1. 可预测的资源标识符：使用 MD5(数字序号) 作为房间标识符，攻击者可轻松枚举全部命名空间。已知规律的确定性哈希不仅没有增加安全性，反而产生了安全假象。

  2. 缺失授权检查：服务端在处理房间资源请求时，未对用户身份或权限进行校验。任何知道 URL 的访问者都可以直接访问任意房间，无论该房间是否在 UI 中可见。

  3. 依赖前端可见性做访问控制：系统仅通过前端页面（图片地图）控制哪些房间可被”看到”，完全信赖客户端无法发现未引用的端点，而未在后端施加任何访问控制。

五、攻击复现步骤

步骤 1: 访问主页 http://10.49.161.165/，查看 HTML 源码。
步骤 2: 发现 13 个 <area> 元素的 href 均为 32 位十六进制字符串。
步骤 3: 识别为 MD5 哈希，并验证 MD5("1") = c4ca4238... 匹配页面中的第一个端点。
步骤 4: 推断存在房间 0，计算 MD5("0") = cfcd208495d565ef66e7dff9f98764da。
步骤 5: 访问 http://10.49.161.165/cfcd208495d565ef66e7dff9f98764da。
步骤 6: 获取 Flag。

复杂度：极低。无需认证、无需工具、仅需浏览器和对 MD5 的基本认知。

六、影响评估

影响维度 评估
机密性 高 — 可访问不应公开的隐藏资源，泄露敏感信息
完整性 无直接影响
可用性 无直接影响
利用难度 极低 — 无需认证，哈希规律可通过简单枚举发现
影响范围 所有未在前端引用的隐藏房间 (0, 14, 15, …)

七、修复建议

7.1 短期修复（高优先级）

  1. 实施服务端访问控制

    • 在后端为每个房间维护授权列表（哪些用户可访问哪些房间）
    • 每次请求时校验当前用户是否有权访问目标房间
    • 无权限时返回 HTTP 403 Forbidden

  2. 使用不可预测的资源标识符

    • 将数字序号替换为随机生成的 UUID（如 UUIDv4）
    • 避免使用可推导的哈希作为资源标识
    • 例如：/room/a1b2c3d4-e5f6-7890-abcd-ef1234567890

7.2 中期改进

  1. 实施基于会话的授权机制

    • 引入用户会话管理
    • 房间访问权限与服务端会话绑定
    • 未登录用户仅能访问公开房间

  2. 添加请求频率限制（Rate Limiting）

    • 对房间端点实施速率限制
    • 防止自动化的批量枚举攻击

7.3 长期架构改进

  1. 采用成熟的授权框架

    • 实施 RBAC (基于角色的访问控制) 或 ABAC (基于属性的访问控制)
    • 所有资源访问统一经过授权中间件

  2. 安全编码规范

    • 建立安全编码指引，明确”默认拒绝”原则
    • 所有新端点默认需要授权检查，公开端点需显式声明
